###############################################################################
# SECTION A — THEME EDITOR · APP APPROVFY EMBED (storefront widget)
###############################################################################
  Open: Theme customize → App embeds → choose **Approvefy**

  Listed options on the embed panel (exact labels):

    A1. Enable Custom Registration Form  
        Checkbox. ON = custom registration / approval workflow active.  
        OFF = default theme registration (no Approvefy form).

    A2. Pending Approval Message  
        Single-line text shown after registration (English default in schema).

    A3. Pending Approval Message (French)  
        French variant when store language fits that use case.

    A4. Form to display  
        Text. **Leave blank** = use Form Builder **default** form.  
        **Paste Form ID** = lock storefront to one form row from Form Builder.

  Then click **Save** on the theme editor.


###############################################################################
# SECTION B — CUSTOMERS PAGE
###############################################################################
  Open: Apps → Approvefy → **Customers**

  --- B0. Toolbar / chrome ---
    · Page title Customers (or analytics header if present).

  --- B1. Tabs (status filter for the whole list) ---
      1) All Customers  
      2) Pending  
      3) Approved  
      4) Rejected  

  --- B2. Search ---
    · Text box: filters results (name/email style search loaded from server).

  --- B3. Date filter ---
    · Opens **Filter by date** modal:  
        • From / To read-only preview fields  
        • Calendar **DatePicker** to pick range  
      Buttons: **Apply** | **Cancel**

  --- B4. Rows per page (page size) ---
    · Choices typical in app code: **25 | 50 | 100 | 200 | All**  
      (“All” uses a capped max for listing — export may differ.)

  --- B5. Pagination ---
    · **Previous / Next** on the footer when paging.

  --- B6. Refresh ---
    · Button with refresh icon reloads latest list/data.

  --- B7. Export CSV ---
    · Opens modal **Export customers**  
      Radio choices:  
        • **Current page**  
        • **All customers** (respects broader query; see CSV route limits)  
        • **Selected: N customer(s)** (disabled until you tick row checkboxes)  
      Buttons: **Export** | **Cancel**

  --- B8. Edit columns ---
    · Opens column editor: toggle visibility per column  
    · Drag to reorder columns (saved in browser local storage)  

        Column keys shown in UI (labels):  
        • Customer name | Email | Company | Phone | Status | Date Join | Action  

  --- B9. Table row actions (per applicant) ---
    · Select checkbox for bulk workflows.  
    · Row icons typically: view/edit path, approve if applicable, deny, delete, etc.
      (exact icon set follows current Polaris deployment.)

  --- B10. Bulk promoted actions (when rows selected) ---
    · **Approve**  
    · **Reject** — disabled when every selected row’s tags already include the  
          literal Shopify tag **`status:approved`** (matches current Customers UI logic)  
    · **Delete customers** — opens destructive flow (see B11).

    Note — Status badges in the table interpret **`status:approved`** vs **`status:denied`** vs pending.  
    Your **Approved customer tag** under Settings → Approval should align with **`status:approved`**  
    unless you customize code; otherwise badges and bulk Reject disabling may mismatch.

  --- B11. Delete customers modal (bulk) ---
    Step 1 — Pick mode:  
        • Delete from Shopify only (app keeps records)  
        • Delete from App server only (Shopify stays)  
        • Delete from Both — permanent; cannot undo  
      **Cancel**

    Step 2 — Confirmation text repeats mode; buttons **Back** | destructive confirm  

  --- B12. Single-row delete ---
    Same three-mode concept as bulk (two-step modal in code).

  --- B13. Detail modal (opening a registration) ---
    Shows when you drill into a registration: identity fields, labels for custom  
    Form Builder data, timestamps, reviewer info when present  
    Actions may include approve / reject from here depending on UX version.

  --- B14. Deny/Reject modal ---
    Follow on-screen prompts to confirm denial (fires rejection email rules if configured).

  EXAMPLES

    · Morning triage: tab **Pending** → Approve/Reject → Export CSV “All” weekly.  
    · Tags: Approved customers mirror **Settings → Approval → Approved tag**.

# SECTION D — FORM BUILDER (INSIDE ONE FORM · EDITOR)
###############################################################################
  Opens from Form Builder table name link or `/app/form-builder?formId=…`  
  New form: wizard starts with chosen type from modal.

  --- D1. Tabs (top of builder) ---
    · **General** — fields & form metadata  
    · **Appearance** — visuals (see SECTION E)

  --- D2. General — form-level options ---
    Typical controls (persist with main save):  

      • **Form name** (internal)  
      • **Form type** — Wholesale registration form | Multi-step form  
           (changing type influences progress bar availability)  

      • **Enabled** checkbox — storefront availability for assignments  
      • **Default form** checkbox — ONLY one shop default recommended for blank embed  

      • **Multi-step only:**  
          ☑ Show progress bar (and % progression on storefront wizard)  

      • **Storefront heading** optional — overrides empty with translation defaults  
      • **Storefront description** optional — subtitle under heading  

      • Bulk save action — label like **Save configuration** / save bar wording in UI  

      • **Discard** — restores last saved snapshot (general + appearance together)

  --- D3. General — Each field row (expand row to edit) ---

      Core controls common to fields:  

        • Drag handle — reorder vertical list (@dnd-kit)  
        • Field **type** picker (types listed in D5) — some types limited to ONE per form  
        • **Label** text  
        • **Placeholder** optional  
        • **Help text** optional tooltip/helper line  
        • **Required** checkbox — Email always required (locked ON)  
        • **Enabled** checkbox — hides field without deleting config  
            (Address bundle: toggling Address enables/disables city/state/etc. chain)  

        Width (some types): **30 | 50 | 100 %**  

        Phone type: **default country dial code** picker  

        Date type: **Date format** picker — matches DATE_FORMAT_OPTIONS list  
          e.g. DD/MM/YYYY, MM/DD/YYYY, ISO dates, spelled-out month formats, weekdays…  

        File upload type:  
          • max files 1–20  
          • max size MB presets 2 · 5 · 10 · 15 · 20 · 25  

        Dropdown / Checkbox group / Radio:  
          • **Options** textarea — one choice per line (blank lines trimmed on save logic)  

        Checkbox-group (multiple):  
          • **minimum selections** (`minRequired` 0–10) enforced when required counts matter  

        Newsletter type: preset label / option text behavior if left blank  

        Heading type: **Heading level** h2 · h3 · h4  

        Multi-step: **Step number** allocation per field (wizard grouping)

      Locked rules (auto in code):  

        • Only one each (single-instance types): Password, Phone, Company, Address,  
          City, State/Province, Zip, Country  

        • **Address**: adding inserts full address bundle (address + city/state/zip/country);  
          removing address removes bundle children together  

      Default seeded fields usually include First Name · Last Name · Email (built-in defaults).

    · **Add field** popover lists only allowed types (duplicate singletons omitted).

    · **Remove**: cannot remove default-system core rows like default first names in some UX;  
      address children not individually deleted.

  --- D5. Complete FIELD TYPE list (exact builder labels → values) ---

      First Name                 (first_name)  
      Last Name                  (last_name)  
      Email                      (email) — required locked  
      Password                   (password)  
      Phone                      (phone)  
      Company                    (company)  
      Address                    (address) — bundle  
      City                       (city)  
      State / Province           (state)  
      Zip / Postal Code          (zip_code)  
      Country                    (country)  
      Text                       (text)  
      Text Area                  (textarea)  
      Number                     (number)  
      Date                       (date)  
      Dropdown                   (dropdown)  
      Multiple Checkbox          (checkbox)  
      Newsletter subscription    (newsletter)  
      Radio                      (radio)  
      Heading                    (heading)  
      File Upload                (file_upload)

    Font family dropdown options:  

      • System (Shopify default)  
      • Arial · Helvetica · Verdana · Tahoma  
      • Open Sans · Roboto · Lato · Montserrat · Poppins

      • Storefront heading size (14–48)  
      • Storefront description size (12–24)  
      • Label size (11–20)

      • Button font size (10–24)  

    **Storefront heading and description alignment**  

      Desktop: Heading alignment Left|Center|Right  
               Description alignment Left|Center|Right  

      Mobile:  Heading alignment Left|Center|Right  
               Description alignment Left|Center|Right  

  --- E4. Colors (hex fields + picker swatches) ---
    • Card background  
    • Card text  
    • Storefront heading (h2 title)  
    • Storefront description (subtitle muted)  
    • Button background  
    • Button text  
    • Input background  
    • Input border  
    • Accent (focus, checkbox, radio highlights)  
    • Error (asterisk & validation messages)  

  --- E5. Radius & spacing ---

    Slide ranges:  

      • Input border radius (0–24 px)  
      • Button border radius (0–40 px capped)  
      • Form max width (320–1500 px)

  --- E6. Custom CSS ---

      Large multi-line textarea.  
      Help: leave empty = only sliders/templates apply — non-empty persists raw CSS overrides.
